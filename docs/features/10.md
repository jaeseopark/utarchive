# PR 10 — Search & Play Count

**Phase:** 2 — Core REST API  
**Depends on:** PR 02, PR 03, PR 04

## Goal

Expose a unified full-text search endpoint across songs, artists, and albums, and a play-count increment endpoint used by the streaming player.

## Endpoints

| Method | Path | Auth | Description |
|---|---|---|---|
| `GET` | `/api/search` | ✅ | Full-text search across all entities |
| `POST` | `/api/songs/:id/play` | ✅ | Increment play count for a song |

## Search

### Query Parameter

`GET /api/search?q=<query string>`

Returns 400 if `q` is missing or empty.

### Implementation

Each entity uses its own `tsvector` GIN index (created in PR 02):

```sql
-- songs: tsvector over title + description + url
-- artists: to_tsvector('english', name)
-- albums: to_tsvector('english', title)
```

The endpoint runs three parallel queries and merges results into a single response:

```ts
{
  songs: Array<{ id, title, artistId, preferred, playCount }>,
  artists: Array<{ id, name, aliases }>,
  albums: Array<{ id, title, artistId, yearReleased }>
}
```

Results within each category are ordered by Postgres `ts_rank` descending. Max 20 results per category.

### Prefix search support

Append `:*` to the last lexeme so partial words match:
```sql
to_tsquery('english', 'miku:*')
```

## Play Count

`POST /api/songs/:id/play` atomically increments `play_count` by 1:

```sql
UPDATE songs SET play_count = play_count + 1 WHERE id = $id
```

Returns `{ playCount: <new value> }` with 200.

Note: the streaming endpoint (PR 17) calls this automatically when a stream begins at byte offset 0. This endpoint exists as an explicit API for edge cases (e.g. external URL plays that don't go through the built-in player).

## Listening Analytics

Capture every user listening session with:
- `song_id`
- `user_id` (from auth session)
- `started_at`
- `ended_at`
- `duration_seconds`
- `playback_percent`
- `seeked` (boolean flag if the user scrubbed)
- `device` / `user_agent` (optional)

Admin controls flush or retain only the last `ANALYTICS_RETENTION_DAYS`.

## Acceptance Criteria

- [ ] `GET /api/search?q=` returns 400
- [ ] `GET /api/search?q=test` returns `{ songs, artists, albums }` even if some categories are empty arrays
- [ ] Results are ranked by relevance, not insertion order
- [ ] Partial query `q=mik` returns results containing "miku"
- [ ] `POST /api/songs/:id/play` increments `play_count` and returns the new value
- [ ] `POST /api/songs/nonexistent/play` returns 404
- [ ] Both endpoints return 401 without a valid session cookie
